## Part 1. 어제 로드맵 각 파트 고도화 (12:15~13:30)

목표: 산출물을 "빈칸 없이"에서 "**검증 가능한 초안**"으로. 어제 설문 피드백(출처·실무·분야 교류)을 반영한 키트 개선분을 적극 활용.

- **출처(citation) 붙이기** — TAM/SAM/SOM·경쟁 가격·CAC 등 외부 숫자는 코치에게 "웹에서 확인해 출처 달아줘"라고 요청. (`evidence-research`) 못 찾으면 `[가정]` 유지. → 어제 대부분 `[가정]`이었던 시장 숫자에 실제 출처 1~2개씩.
- **도메인별 정조준** — `docs/도메인별-체크포인트.md`에서 내 아이디어 유형(마켓플레이스/B2B/딥테크/소비자앱/규제민감/커리어)을 골라 "반드시 검증"을 BMC·세일즈·레드팀에 반영.
- **약한 가정 공격** — `vc-red-team-review`로 가장 약한 가정 Top 3를 다시 때리고, "지금 당장 검증할 것"을 행동 단위로.
- **DOCX로 내보내 수정** — `bash scripts/export-docx.sh roadmap "<이름>"` → Word에서 직접 다듬기.

> 강사 멘트: "오늘 빌드할 한 조각을 정하려면, 먼저 **어느 가정이 가장 위험한지**를 알아야 합니다. 그걸 검증할 가장 빠른 실험이 오늘의 랜딩/MVP입니다."

## Part 2. 바이브코딩 빌드 (14:00~16:00) — `vibe-build` 스킬

> **2단계로 분리:** ① `vibe-build`로 **빠르게 배포**(못생겨도 OK, 핵심·waitlist 작동 확인) → ② `design-polish`로 **레퍼런스 기반 디자인**(shadcn/ui, "AI 티" 제거 · `docs/디자인-가이드.md`). 참가자는 "배포됐다!" → "예뻐졌다!" 성취를 두 번 맛본다.

### 도구/스택 (AI-native)
**GitHub + Vercel + Supabase**, Next.js. 참가자는 **로그인(auth)만**, 나머지(레포·스캐폴딩·DB·배포)는 Codex가 CLI로.
```bash
gh auth login        # GitHub
vercel login         # 필요시 npm i -g vercel
supabase login       # 필요시 npm i -g supabase
```

### 트랙 (택1)
| 트랙 | 누가 | 반드시 |
|---|---|---|
| 🅰 **Landing** (기본) | 전원 / 다음 단계가 수요 검증·인터뷰인 사람 | **waitlist 수집 필수** (이메일 → Supabase) |
| 🅱 **MVP** (야심) | 핵심 가치가 "직접 써봐야" 증명되는 사람 | 핵심 기능 **1개 흐름**만 |

> 강사 결정 규칙(10초 안내): "기본은 Landing. **앱을 눌러봐야 가치가 증명되면** MVP. MVP도 흐름 1개로 좁히고, 복잡하면 한 부분만 진짜·나머지는 가짜(하드코딩)."

### 스택 선택 (모르는 사람 많으니 가이드)
- **추천·기본 = Vercel + Supabase** (웹/앱/마켓플레이스/SaaS 대부분)
- 이미 정한 스택 있음 → 그걸로 (오늘은 데모 가능한 범위만)
- 정적만 원해도 **waitlist는 Supabase로**

### 진행 단계
1. **(10분) 트랙·스택·범위 선택** — 어제 `05_개발로드맵`의 MVP 범위에서 "오늘 만들 1개"
2. **(5분) 로그인 체크** — gh / vercel / supabase (사람은 여기까지만)
3. **(10분) 스캐폴딩 + 빈 페이지 먼저 배포** — URL 확보(이후 자동 재배포)
4. **(60~90분) 바이브코딩** — 어제 `01_아이디어정의서`·`03_GTM` 카피 주입. Landing은 waitlist→Supabase(RLS)
5. **(10분) 데모 리허설**

> 강사 강조: **waitlist 없는 랜딩은 의미 없다**(수요 신호가 목적). Supabase 테이블은 **항상 RLS**(익명 INSERT만). 막히면 흐름 유지가 우선.

## 데모 라운드 (16:00~16:45)
1인 60~90초: ① 누구의 무슨 문제 ② 화면에서 1개 동작/waitlist 신청 시연 ③ **이걸로 검증할 가설 1개**(예: "이 메시지로 방문자 중 N%가 waitlist 신청"). 데모 후 URL을 동료·잠재고객에게 실제 공유.

## 트러블슈팅 퀵 (`docs/troubleshooting.md` #12·#13)
- 배포 후 폼이 안 되면 → `vercel env`에 `NEXT_PUBLIC_SUPABASE_URL`·`..._ANON_KEY` 있는지 확인, 추가 후 `vercel --prod`.
- waitlist 저장 안 되면 → 마이그레이션 적용(`supabase db push`)·**익명 INSERT 정책** 확인. anon key만 사용(서비스키 노출 금지).
