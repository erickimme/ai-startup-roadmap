## 느린 참가자 처리

느린 참가자는 완성도보다 빈칸 없는 초안을 목표로 합니다. 숫자가 없으면 `[가정]`이라고 표시하게 하고 다음 단계로 넘깁니다.

## 빠른 참가자 처리

빠른 참가자(쉽다/지루하다고 느끼는 분)는 아래 심화 루프로 보냅니다.

1. 합성 소비자 패널 5명 생성 (`korean-synthetic-consumer`)
2. 메시지 테스트 · 반대 논리 맵
3. 실제 인터뷰 질문 7개 생성
4. `evidence-research`로 핵심 숫자(TAM·가격·CAC) 근거 1~2개 보강
5. `vc-red-team-review` 심화 + 최종 피치 수정
6. **MVP를 한 단계 더**: `startup-cto-roadmap`로 화면 흐름·기술 결정까지, 가능하면 작은 프로토타입/랜딩 시도
7. **실무 위임 연습**: 메인 가이드의 "창업자에게 유용한 Harness 아이디어"/gstack 역할 위임 체크리스트로 내 업무 자동화 1개 설계

## Day 2 빌드 트랙 (vibe-build)

Day 2 후반은 어제 로드맵을 **실제 동작하는 것**으로 만들어 데모한다. `skills/vibe-build/SKILL.md` 참고.

- **도구/스택:** Codex CLI + GitHub·Vercel·Supabase·Next.js. 참가자는 `gh/vercel/supabase login`(**auth만**), 나머지는 에이전트가 CLI로.
- **트랙(택1):** 🅰 Landing(전원·**waitlist 필수**) / 🅱 MVP(핵심 기능 1개 흐름). 결정 규칙: 기본은 Landing, 앱을 직접 눌러봐야 가치가 증명되면 MVP, MVP도 흐름 1개로 좁힌다.

| 시간 | 내용 |
|---|---|
| 0:00-0:10 | 트랙·스택·범위 선택 (어제 `05_개발로드맵`에서 "오늘 만들 1개") |
| 0:10-0:15 | 로그인 체크 (gh / vercel / supabase) |
| 0:15-0:25 | 스캐폴딩 + 빈 페이지 먼저 Vercel 배포(URL 확보) |
| 0:25-1:25 | 바이브코딩 — 어제 카피 주입, Landing은 waitlist→Supabase |
| 1:25-1:35 | 데모 리허설 (60~90초 + 검증 가설 1개) |
| 1:35- | 공개 데모 라운드, URL 공유 |

- 강사 강조: **waitlist 없는 랜딩은 의미 없음**, Supabase는 **항상 RLS**, 막히면 한 조각만 진짜·나머지 가짜로 흐름을 유지한다.
- **빌드 → 디자인 2단계 분리:** `vibe-build`로 먼저 배포(못생겨도 OK)해 성취를 맛본 뒤, `design-polish`로 레퍼런스 기반 디자인을 올린다("AI 티" 제거 · `docs/디자인-가이드.md`). 전원 배포 후 다 같이 디자인 라운드.
- **어제 이어서(중요):** 참가자는 어제 clone한 폴더에서 **`git pull`로 하네스를 먼저 최신화**해야 Build·Polish 스킬을 쓸 수 있다(어제 이후 추가됨). `roadmap/`은 보존된다. 안내: "git pull로 최신화 → roadmap/ 이어서 다음 단계부터".
